# AI Marketing OS — Future Roadmap

> Этот документ фиксирует требования к будущему развитию системы.
> Текущий утверждённый Workflow и ядро системы не затрагиваются.
> Ни одно из требований ниже не реализовано в текущей кодовой базе.

---

## Текущий утверждённый Workflow

```
Workflow
  → Research
  → Candidate Ideas
  → Review
  → Idea Bank
  → Create
  → Publication
  → Results
  → Insights
```

Этот цикл считается стабильным и не изменяется до завершения первого реального запуска.

## Архитектурное направление: Campaign Blueprint

> Зафиксировано: 2026-07-02. Не реализовано. Определяет направление развития архитектуры.

### Проблема текущей модели

Текущий Create получает одну идею и создаёт контент. Это слишком упрощённая модель: она не описывает, зачем создаётся контент, куда ведёт пользователя и какую роль играет каждая площадка.

### Целевая последовательность

```
Research
  → Campaign Blueprint
  → Review
  → Create (для каждой площадки по Blueprint)
  → Publication
  → Results
  → Insights
```

### Что такое Campaign Blueprint

Campaign Blueprint — новый логический уровень между Research и Create. Его задача — не создавать контент, а описывать маркетинговую кампанию как единое целое.

Blueprint содержит:

| Элемент | Описание |
|---|---|
| Цель кампании | Что должна достичь кампания в целом |
| Маршрут пользователя | Как пользователь движется между площадками |
| Роль каждой площадки | Функция площадки в воронке |
| Материалы к созданию | Список единиц контента с назначением каждой |
| Следующее действие | Что должен сделать пользователь после каждого этапа |
| Ожидаемый результат | Конечная конверсия или изменение состояния пользователя |

### Ключевой принцип

Контент для разных площадок — не копии одного материала. Каждая площадка выполняет собственную маркетинговую функцию:

| Площадка | Функция в воронке |
|---|---|
| Instagram | Привлечение внимания, охват |
| Pinterest | Поисковый трафик, обнаружение |
| Статья / Дзен | Объяснение, доверие, SEO |
| Telegram | Удержание, закулисье, прогрев существующей аудитории |
| Сайт | Конверсия, финальное действие |

### Влияние на архитектуру

- Create перестаёт быть привязан к одной идее — он работает с Blueprint как источником задания.
- Blueprint становится единицей планирования: одна кампания = один Blueprint = несколько единиц контента.
- Review после Research проверяет не идеи, а Blueprint: правильно ли определены цель, маршрут, функции площадок.
- Idea Bank трансформируется в Campaign Bank.
- Results собирают данные по кампании в целом, а не по отдельной публикации.

## Архитектурное решение: Campaign Specification

> Зафиксировано: 2026-07-02. Не реализовано. Принято как проектное решение.

### Что такое Campaign Specification

Campaign Specification — единый машиночитаемый документ, который создаётся после утверждения кампании и становится единственным источником данных для Create, Publisher, Analytics и других будущих модулей. Модули не принимают самостоятельных решений — они читают Specification и работают по нему.

Формат хранения: JSON (машиночитаемый источник правды) + MD (человекочитаемый вид).
Расположение: `data/{project}/campaigns/campaign_{id}.json`

### Структура

**Блок: Идентификация** — `id`, `created_at`, `status`, `project`

**Блок: Источник** — связь с Research-сессией и идеей из Idea Bank (`idea_title`, `approved_at`)

**Блок: Кампания** — маркетинговая логика: `goal`, `funnel_stage`, `target_action`, `target_result`, `user_journey`

**Блок: Площадки** — массив `platforms`, каждый элемент описывает одну единицу контента:
- `platform`, `format`, `role`, `funnel_function`
- `key_message`, `cta`, `next_destination`
- `tone`, `visual_direction`, `restrictions`
- `content_status` (`pending` → `created` → `published`)
- `content_file`, `published_at`, `published_url`

**Блок: Бренд** — `voice`, `restrictions` (применяются ко всем площадкам)

### Жизненный цикл

```
Blueprint утверждён пользователем
    → Specification создаётся автоматически (status: approved)
    → platforms[*].content_status = "pending"

Create
    → platforms[i].content_status = "created"
    → platforms[i].content_file = путь к файлу

Publisher
    → platforms[i].content_status = "published"
    → platforms[i].published_at, published_url

Все площадки published
    → campaign.status = "completed"

Analytics / Results
    → читает completed Specification
```

### Принятые архитектурные решения (2026-07-02)

**1. Idea Bank не заменяется.**
Idea Bank остаётся текущим рабочим объектом. Campaign Specification создаётся как следующий объект после утверждённой идеи. Миграция Idea Bank → Campaign Bank — отдельный будущий шаг, не часть текущей реализации.

**2. Specification создаётся автоматически.**
Пользователь утверждает стратегию кампании (Campaign Blueprint). После утверждения Specification генерируется системой без ручного заполнения. Пользователь не заполняет Specification вручную.

**3. Одна идея = одна кампания (MVP-правило).**
Одна утверждённая идея порождает ровно одну Campaign Specification — одну кампанию дня. Одна идея не может порождать несколько параллельных кампаний. Это ограничение MVP; пересмотр возможен после накопления реальных данных.

### 1. Multi-platform Create

**Суть**
Один утверждённый результат Research / Idea Bank используется как единая маркетинговая основа для генерации контента под разные площадки.

**Принцип**
Маркетинговая основа (идея, триггер, адаптация, этап воронки) создаётся один раз — на этапе Research. Каждая площадка получает собственный формат подачи, но все версии контента строятся вокруг одной основы.

**Площадки**

| Площадка | Формат |
|---|---|
| Instagram | Reels, карусель, пост |
| Telegram | Текстовый пост, статья |
| Threads | Короткий текст, тред |
| Дзен | Статья / нарратив |
| Pinterest | Описание + визуал |
| Сайт | SEO-статья |

**Текущее состояние**
Create (Этап 7) генерирует единый пакет публикации без привязки к конкретной площадке. Разделение по площадкам — следующий шаг после накопления реальных данных о результатах.

**Что потребуется в реализации**
- Расширить JSON-схему Create: добавить поле `platform` или генерировать отдельный блок для каждой площадки.
- Либо создать отдельный модуль `engine/adapt.py`, принимающий базовый пакет и платформу.
- Конфигурацию активных площадок хранить в `config.yaml` на уровне проекта.

### 2. Goal + Next Action (маршрут пользователя)

**Суть**
Каждая единица контента создаётся не как отдельная публикация, а как часть маркетингового маршрута пользователя.

**Для каждой публикации определяется**

| Атрибут | Описание |
|---|---|
| `goal` | Цель публикации (охват / доверие / заявка / повторная продажа) |
| `role` | Роль в воронке (знакомство / прогрев / конверсия) |
| `cta` | Призыв к действию |
| `next_action` | Что должен сделать пользователь после просмотра |
| `next_destination` | Куда переходит пользователь (ссылка / профиль / мессенджер) |
| `funnel_position` | Место публикации в общей воронке проекта |

**Связь с текущей архитектурой**
Этап воронки (`funnel_stage`) уже определяется в Research и передаётся в Create. `goal`, `role`, `next_action`, `next_destination` и `funnel_position` — расширение этой логики на уровне пакета публикации.

**Что потребуется в реализации**
- Добавить эти поля в JSON-схему Create (`engine/create.py`).
- Добавить `funnel_map` в профиль проекта (`data/<project>/profile.md`) — описание полного маршрута пользователя для данного проекта.
- Передавать `funnel_map` в системное сообщение через `context.py`.

### 3. Results + Insights self-learning loop

**Суть**
Results собирают данные по опубликованному контенту. Insights находят закономерности и передают их в следующий Research, чтобы система постепенно улучшала качество контента на основе собственных данных.

**Данные, которые собирает Results**

| Метрика | Источник |
|---|---|
| Охват | Платформа |
| Сохранения | Платформа |
| Комментарии | Платформа |
| Переходы / клики | Платформа |
| Заявки | CRM / вручную |
| Площадка | `config.yaml` |
| Формат | пакет публикации |
| Тема / идея | `idea_bank.json` |
| CTA | пакет публикации |

**Закономерности, которые ищет Insights**
- Какие темы дают лучший охват / сохранения / заявки.
- Какие форматы работают лучше на каждой площадке.
- Какие CTA дают больше переходов / заявок.
- Какие площадки лучше подходят для конкретных тем.
- Какие маршруты пользователя дают конверсию.

**Принцип self-learning**
Insights не заменяют Research — они обогащают системное сообщение перед следующим Research. `insights.md` уже передаётся в LLM через `context.py`. Сейчас он заполняется вручную. В будущем — частично автоматически на основе `results_log.md`.

**Что потребуется в реализации**
- Создать модуль `engine/insights.py`: принимает `results_log`, вызывает LLM для анализа, обновляет `insights.md`.
- Добавить команду `python engine/run.py insights` в `run.py`.
- Добавить шаг Insights в `workflow.py` после получения результатов.
- Структурировать `results_log.md` так, чтобы данные можно было агрегировать (текущая табличная структура уже подходит).

### 4. Reels music — region requirement

**Суть**
При генерации Reels система должна учитывать регион публикации и предлагать только музыку, доступную в этом регионе.

**Принцип**
Регион публикации — параметр конфигурации проекта, а не жёсткое правило в коде. Разные проекты могут иметь разные регионы.

**Текущее состояние**
Create (Этап 7) уже генерирует `music_mood` — тип и настроение аудио без указания конкретных треков. Это сделано намеренно: выбор конкретной музыки и проверка региональной доступности отложены до реализации этого требования.

**Параметр конфигурации**
```yaml
# config.yaml
projects:
  active: cleaning

project_settings:
  cleaning:
    publication_region: "RU"
```

**Для текущего проекта**
Регион публикации — Россия (`RU`). При подборе музыки для Reels учитывать доступность треков в российском сегменте платформ.

**Что потребуется в реализации**
- Добавить `publication_region` в `config.yaml` на уровне проекта.
- Передавать регион в промпт Create при генерации `music_mood` и `reels_script`.
- В будущем: интеграция с API платформ для проверки доступности конкретных треков по региону.
